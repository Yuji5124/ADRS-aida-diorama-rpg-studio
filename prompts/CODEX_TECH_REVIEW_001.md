## 前提知識（ADRS の方針）

- ADRS は **ドット絵タイル + 3D ジオラマ表現** で短編 RPG を制作するツール。
- Demo 01 は「機能網羅」ではなく **「短い RPG がタイトルからクリアまで一周遊べる縦切り」** が成功条件。
- **イベント統一が ADRS の心臓部:** 会話・宝箱・ワープ・戦闘開始を個別機能にせず、
  **すべて Event の `commands`（6 コマンド）で表現** する。
  - 6 コマンド: `message` / `addItem` / `setFlag` / `transfer` / `startBattle` / `clearGame`
- **「HD-2D」という名称は使わない。** 「ピクセルジオラマRPG / Pixel Diorama RPG / Diorama RPG Studio」を使う。

## レビュー観点（必須 10 項目）

各観点について、**具体的な根拠（どのファイル・どの記述か）** を挙げて指摘してください。

1. **縦切りの技術的成立性** — Demo 01 の一周（タイトル→村→会話→宝箱→洞窟→通常戦→ボス→クリア）が技術的に成立するか。
2. **データモデルの不足・矛盾** — `11-data-model.md` の各エンティティに不足・矛盾・参照切れがないか。
3. **6 コマンドで台本が成立するか** — `40-demo01-scenario.md` が本当に 6 コマンドだけで表現できているか。隠れて必要なコマンドがないか。
4. **`startBattle` 後の続行仕様の危険性** — 戦闘後にイベントの続きコマンドへ戻る仕様（特にボスの `startBattle` → `setFlag` → `message` → `clearGame`）が曖昧／危険でないか。勝敗・逃走・全滅時の分岐はどう扱うべきか。
5. **flag 設計の不足** — `chest_potion_opened` / `boss_defeated` の 2 flag で破綻しないか。変数なしで足りるか。
6. **戦闘・ジョブ・スキルで先に決めるべき点** — `22-battle-spec.md` / `23-job-skill-spec.md` 着手前に確定すべき仕様（行動順、ダメージ式、対象選択、MP 管理、勝利処理など）。
7. **React + TypeScript + Three.js 想定時のリスク** — もしこの技術構成を採るとした場合の技術リスク（状態管理、グリッド↔3D座標変換、描画とゲームロジックの分離、データ駆動の更新など）。**ただし技術選定は確定しないこと。**
8. **実装前に作るべき追加仕様書** — 現状の docs に足りない仕様書（例: アーキテクチャ、座標系、戦闘詳細、セーブ無し前提の状態管理など）。
9. **Demo 01 で後回しにすべき機能** — スコープに紛れ込んでいる、または後回しが妥当な機能。
10. **20〜30 ターン完成のための優先順位** — `41-roadmap.md` を踏まえ、着手順序の妥当性と組み替え提案。

### 特に重点的に見てほしい点

> **イベントシステムとデータモデルの矛盾** を最優先で見てください。
> ADRS は会話・宝箱・ワープ・戦闘開始を個別機能にせず **すべて Event の commands で統一** する方針です。
