# 24 - ジオラマ表現方針 (Diorama Rendering)

> ADRS Demo 01 における **「ピクセルジオラマRPG」表現の上限とルール** を固定する。
> このドキュメントの目的は、`10-architecture.md`（技術選定）に入る前に **描画仕様をぶらさない** こと。
> 「作った瞬間に見栄えする」を **最小コスト** で達成する方針。

> ⚠️ ADRS では **「HD-2D」という名称は使わない。** 本書でも「ピクセルジオラマRPG / Pixel Diorama RPG / Diorama RPG Studio」を使う。

---

## 基本思想

- 見栄えは **ライト1灯 + 簡易影 + 固定斜め見下ろし** で達成できる、とみなす。
- 表現に凝るより、**Demo 01 が一周遊べる縦切り** を優先する（→ `01-demo01-scope.md`）。
- 描画は **ゲームロジックに影響しない**。影・発光・スプライト表現は見た目だけの責務。
- 凝った演出（霧 / DOF / 天候 / 複数光源 / リアルタイム影必須化）は **後続デモ** に回す。

---

## 1. カメラ

Demo 01 は **固定斜め見下ろし**。

| 項目 | 方針 |
|---|---|
| カメラ位置 | 固定（斜め見下ろし） |
| 回転 | **なし**（自由回転させない） |
| プレイヤー操作 | カメラ操作は入れない |
| 投影 | **正投影風 / アイソメトリック寄り** |
| 追従 | 主人公を追従してよい。ただし **角度は変えない**（平行移動のみ） |
| カメラ角度（仮） | 水平方向 45°、俯角 35〜45° 程度 |

- 角度の正確な数値は実装時に微調整してよい。**Demo 01 では自由回転は禁止**。
- ズームも Demo 01 では固定でよい（可変ズームは後回し）。

---

## 2. タイルの立ち上げ規則

2D タイルを 3D ジオラマとして表示するための **最小規則**。Demo 01 では **自由な高さ編集はしない**。

### 基本方針

- マップは **グリッドベース**。**1 タイル = 1 セル**。
- 地面タイルは **薄い板** として表示。
- 壁・崖・岩などは **高さを持つブロック** として表示。
- 水は地面より少し低く、または平面として表示。
- 道・草・洞窟床は **同じ高さ**。
- 家・木・宝箱・NPC・敵シンボルは **オブジェクト** としてタイル上に配置。

### 高さの目安（段階式）

| タイル / 種別 | 高さ | 表示 |
|---|---|---|
| `water` | 低い平面 | 地面より少し低い、または平面 |
| `grass` / `road` / `cave_floor` | 標準高さ（基準=0） | 薄い板 |
| `wall` / `rock` / `cliff` | 高さ 1 段 | 高さを持つブロック |
| `house` / `tree` | オブジェクト扱い | タイル上に立てる |
| `chest` / `npc` / 敵シンボル | オブジェクト扱い | スプライト（§5） |

- 高さは **段階式（離散）**。連続的な高さ・傾斜・自由編集は **Demo 01 では行わない**。
- **段差移動・高低差ルート探索は扱わない**（移動は `11-data-model.md` の `collision` に従う平面グリッド）。
- `cliff` は段階高さの一例として定義に含めるが、Demo 01 の台本（`40`）では未使用でもよい。

---

## 3. ライト

Demo 01 は **ライト1灯** を基本。

| 項目 | 方針 |
|---|---|
| メインライト | **1 灯**（方向性ライト想定、`11-data-model.md` の `diorama.light` に対応） |
| 方向 | 斜め上から |
| 環境光 | 暗くなりすぎない最低限の環境光相当は **許容**（影部分が真っ黒にならない程度） |
| 動的ライト | **複数の動的ライトは入れない** |
| 松明・窓の光 | **見た目上の発光表現に留める**（自発光スプライト / グロー）。**実ライトとしては扱わない** |

- `11-data-model.md` の `Map.diorama.light`（`type: directional`, `intensity`, `angle`）と整合させる。

---

## 4. 影

Demo 01 は **擬似影を優先**。

| 項目 | 方針 |
|---|---|
| 既定 | **擬似影**（丸影 / 薄い影をオブジェクト直下に置く） |
| リアルタイム影 | **必須にしない** |
| 対象 | キャラ・宝箱・木・家などの下に簡易影 |
| ゲームロジック | 影は **ロジックに影響しない**（当たり判定・trigger と無関係） |
| 性能 | 影が重くなる実装は避ける |
| 将来 | リアルタイムシャドウは **後続デモの候補** |

- `11-data-model.md` の `Map.diorama.shadow: true` は「簡易影を出す」の意味と解釈する。
- リアルタイム影に切り替える場合も、データ構造（`shadow` フラグ）は流用できるようにしておく。

---

## 5. スプライト / オブジェクト表現

Demo 01 では、ドット絵キャラ・NPC・敵シンボルは **2D スプライトを 3D 空間に立てて** 表示する。

| 対象 | 表現 |
|---|---|
| 主人公 / NPC / 敵シンボル | **板ポリゴン風スプライト**（カメラ方向に見やすい向き） |
| 宝箱 / 看板 / 草花 | タイル上の **オブジェクト**（スプライト or 簡易立体） |
| 家 / 木 | **簡易 3D または スプライト的表現のどちらでもよい** |
| アニメーション | **複雑なアニメーションは不要**（待機・歩行程度に留める。なくてもよい） |

- スプライトの向き（ビルボード）方針: カメラ角度が固定なので、**固定角度に正対** させれば足りる（フルビルボード必須ではない）。
- `appearance.tile` がスプライト/オブジェクトの見た目を指す（`21-event-system-spec.md` の Event.appearance と対応）。

## 6. `visible:false` の描画ルール

> `EventPage.appearance.visible === false`（または有効ページが無い）の場合のルール。
> **`11-data-model.md` / `21-event-system-spec.md` の定義と一致させる**（描画側の再掲＋具体化）。

### Play Mode

| 項目 | 挙動 |
|---|---|
| 描画 | **しない**（メッシュ/スプライトを出さない） |
| 影 | **描画しない**（擬似影も出さない） |
| 当たり判定 | **なし**（踏み抜けられる） |
| trigger | **発火しない** |
| 選択 | 選択対象にならない |

> これは `21-event-system-spec.md` の「非表示・当たり判定なし・trigger 発火なし」と完全に一致する。
> 例: 通常敵シンボルは勝利後ページで `visible:false` → **描画も影も消え、再接触も起きない**。

### Editor Mode

- イベント一覧には **残る**（消滅したように見えても、データとしては存在し編集可能）。
- 「非表示イベントを表示」オプションを将来追加する **余地は残す**（薄表示やアイコン表示など）。
- **Demo 01 では通常、Editor で非表示イベントを描画しなくてよい**（一覧に残れば足りる）。

## 10. 次の `10-architecture.md` で決めるべき技術観点

> 本書で描画の **上限とルール** は固定した。次は実装前の全体設計でこれらの **責務分担** を決める。

1. **レンダリング層とゲームロジック層の分離** — 描画は状態を読むだけ、ロジックは描画に依存しない、という境界の引き方。
2. **Map JSON → 描画オブジェクトへの変換責務** — タイル配列・高さ規則から立体メッシュ/板/スプライトを生成するのは誰か。
3. **`Event.visible` と 描画 / 衝突 / trigger の同期** — flag 変化 → ページ再評価 → visible 反映を、3 系統（描画・当たり判定・trigger）に一貫適用する仕組み。
4. **タイル座標 ↔ ワールド座標の変換** — グリッド (x,y) と 3D ワールド座標・高さの相互変換の置き場所。
5. **カメラ追従の責務** — 主人公追従（平行移動のみ・角度固定）をどの層が担うか。
6. **擬似影の生成責務** — オブジェクト配置時に影を自動付与する仕組みと、`visible:false` 時に影も消す連動。
7. **Editor Mode と Play Mode の描画差分** — 非表示イベントの扱い、グリッド/ガイド表示など、2 モードの描画分岐の設計。
8. **発光表現（松明・窓）の扱い** — 実ライトにせず自発光/グローで見せる方針を、どこで適用するか。

> ここが固まれば、実装前ラストの `10-architecture.md`（全体設計）に入れる。
