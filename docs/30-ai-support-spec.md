# 30 - AI サポート仕様 (AI Support)

> ADRS Demo 01 の AI サポート機能を確定する。**本物の AI 実装ではなく**、
> **入出力・承認フロー・`Event.commands` への反映ルール**を固定する。
> AI サポートは **Editor Mode で制作を助ける補助機能**であり、ゲーム実行に必須の AI ではない。

---

## 1. AI サポートの位置づけ

- ADRS Demo 01 の AI サポートは、**ゲーム実行エンジンではなく、Editor Mode で制作者を助ける補助機能**である。
- **AI サポートが無くても、ADRS Demo 01 は成立する**（一周クリアできる）。
  - AI は `message` の文章候補を提案するだけ。ゲーム進行（flag / 戦闘 / 遷移）には関与しない。
- `10-architecture.md` の AI Support レイヤー（Editor 専用・Play 非依存）と一致する。

---

## 2. Demo 01 で扱う AI 機能

Demo 01 では **1 機能だけ**:

```text
NPC セリフ・クエスト文の提案
```

対象例:
- 村人の初回会話
- 洞窟前の老人の警告
- ボス前の警告文
- 宝箱発見時の短いメッセージ
- クリア後の一言

**Demo 01 の最優先対象**（→ `40-demo01-scenario.md` §6 と整合）:
- `ev_npc_villager`（村人 / 世間話）
- `ev_npc_oldman`（洞窟前の老人 / クエスト提示）
- `ev_boss`（ボス前の警告）

> それ以外（宝箱メッセージ・クリア後の一言）は対象に **してもよい**が、Demo 01 では上記 3 つを優先。

---

## 3. 入力項目

AI サポートパネルでユーザーが指定できる項目。**基本は選択式**（自由入力も許可してよいが簡単に）。

| 入力 | 選択肢（例） |
|---|---|
| **用途** | 村人の会話 / 洞窟前の警告 / ボス前の警告 / クエスト文 / クリア後メッセージ |
| **キャラクター種別** | 村人 / 老人 / ボス / ナレーション |
| **口調** | やさしい / まじめ / 少し不思議 / 短く |
| **状況メモ**（自由入力） | 例: 「洞窟に魔物がいることを伝える」 |
| **文字数目安** | 短い / 普通 |

> 状況メモのみ自由入力。他は選択式で素早く指定できるようにする。

---

## 4. 出力形式

AI サポートは **必ず 3 案**を出す。

```text
候補1：「洞窟の奥で、変な光を見たんだ。」
候補2：「あの洞窟には近づかない方がいいよ。」
候補3：「勇気があるなら、東の洞窟を見てきてくれ。」
```

各候補に付けるもの:
- 候補番号
- セリフ本文
- **採用ボタン**
- 再生成ボタン（あってよい）
- 手動編集欄（あってよい）

---

## 5. 採用フロー（最重要）

> **AI 出力は自動反映しない。** 必ず人間が採用ボタンを押した案のみ反映する。

1. ユーザーが **用途・キャラ・状況**を選ぶ
2. AI サポートが **3 案を提示**
3. ユーザーが **1 つを選ぶ**
4. 必要なら **手動編集**する
5. **「採用」ボタン**を押す
6. **対象 Event の `message` command の `text` に反映**する

このとき:
- **新しい Event command は追加しない**。
- 反映先は **既存の `message` command**。

```jsonc
{ "type": "message", "text": "洞窟の奥で、変な光を見たんだ。" }
```

---

## 6. Event.commands との関係

> AI サポートは **Event.commands 統一方針を崩さない**。

- AI が作るのは **`message.text` の候補だけ**。
- 以下の commands の **追加・削除・順序変更を Demo 01 では AI に任せない**:
  - `addItem` / `setFlag` / `transfer` / `startBattle` / `clearGame`
- **Demo 01 では、AI はイベント構造を変更しない。AI は文章候補だけを提案する。**

> つまり AI は「下書き支援」。イベントの骨格（commands の並び・条件・遷移）は人間が組む。

---

## 7. 擬似 AI・テンプレート生成方針

> Demo 01 では **実 AI API 接続を必須にしない**。

### Demo 01
- **擬似 AI・テンプレート生成でよい**。
- 用途・キャラ・口調・状況メモをもとに、**ルールベースで 3 案**を出す。
- **インターネット接続や API キーがなくても動く**想定。

### 後続デモ
- 実 AI API 接続を検討。
- 外部 API を使う場合は **API キー管理・失敗時フォールバック・利用規約・コスト管理**が必要。
- **Demo 01 では未対応**。

---

## 8. Editor Mode / Play Mode の扱い

| 項目 | Editor Mode | Play Mode |
|---|---|---|
| AI サポート表示 | する | しない |
| セリフ提案 | できる | できない |
| 採用・反映 | できる | できない |
| Event.commands 実行 | しない | する |
| AI API 依存 | なしで成立 | 依存しない |

> **AI サポートは Editor Mode 専用**。Play Mode（ゲーム実行）には一切関与しない。
> Play Mode は、AI が編集済みの `message` を **ただ実行するだけ**（AI 呼び出しは起きない）。

---

## 9. 失敗時フォールバック

> AI サポートが失敗しても **制作を止めない**。

- 候補生成に失敗したら **テンプレート文を表示**する。
- ユーザーが **手動入力**できる。
- **採用せず閉じ**られる。
- **Play Mode には影響しない**。
- 実 AI API が未接続でも **擬似 AI で動く**。

## 12. データへの反映ルール

AI 提案を採用したとき、変わるのは **Project Data 内の対象 Event の `message.text` のみ**。

- **Project Data** 内の対象 Event の `message` command の `text` が更新される。
- **GameState は更新しない**（実行時状態に触れない）。
- **Battle Runtime は関係しない**。
- **Renderer は更新後の Event データを表示するだけ**。
- **Play Mode では、更新済みの `message` command が実行される**。

**Before**
```jsonc
{ "type": "message", "text": "洞窟には気をつけて。" }
```

**After**
```jsonc
{ "type": "message", "text": "洞窟の奥で、変な光を見たんだ。" }
```

> 変わるのは `text` の中身だけ。command の `type` も、commands の並びも変えない。

---

## 13. セーフティ・品質ルール

> Demo 01 の範囲で。

- 長すぎる文章は避ける。
- 暗すぎる・過激すぎる内容は避ける。
- 子どもでも読める程度の **短い文**を優先。
- ADRS の世界観に合う、**やさしく短い文**を優先。
- **採用前に人間が確認する**。
- **自動反映しない**。
- 出力が気に入らなければ **手動編集できる**。
